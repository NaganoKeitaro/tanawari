# 詳細設計書 (Detailed Design Document)

**作成日**: 2026-01-09
**最終更新日**: 2026-03-13
**プロジェクト**: 棚割管理システム (Planogram System)

## 1. プログラム設計書 (Logic Design)

本システムの中核となる「個店棚割自動生成ロジック」の詳細を定義する。

### 1.1 共通前提（座標管理とシステム）
*   **単位**: システム全体で `mm`（ミリメートル）に統一。1尺 = 300mm（`SHAKU_TO_MM = 300`）。
*   **店舗のフロア配置（2Dグリッド平面図）**: `StoreFixturePlacement` によりフロア左上を原点とする `positionX`, `positionY` (mm) と `direction` (0/90/180/270度) の絶対座標で管理される。
*   **棚割空間の配置（立面図/平面図）**: 左端を原点とし、商品は `positionX`（左端からの相対距離 mm）と `shelfIndex`（段数、0-indexed）で管理される。
*   フロア配置と棚割空間配置は `什器 (Fixture)` を軸として結合され、画面上で適切に紐づけて表示される。
*   **入力**: 標準棚割（商品リスト、ブロック配置）、店舗棚サイズ（幅 mm）、商品売上ランク
*   **出力**: 個店棚割（商品リスト、配置、メタデータ）
*   **ビジュアル表示スケール**: `SCALE = 0.3`（1mm = 0.3px）

### 1.2 ロジックA: カット (Cut)
店舗棚幅 < 標準棚幅 の場合に適用される縮小ロジック。重要度の低い商品から削除する。

**処理フロー** (`automationService.ts` / `applyRuleA`):
1.  **判定**: `targetWidth < standardWidth`
2.  **フェイス削減ループ**:
    *   標準棚割内の全商品を対象とする。
    *   売上ランク順（降順：ランク数値が大きい＝売上が低い順）にソート。
    *   各商品のフェイス数が2以上であれば、1になるまで減らす。
    *   削減の都度、総幅を再計算し、`targetWidth` に収まれば終了。
3.  **商品削除ループ**:
    *   フェイス削減でも収まらない場合、売上ランクの低い順（数値が大きい順）に商品を完全削除（`isCut=true`）する。
    *   削除の都度、総幅を再計算し、収まれば終了。
4.  **再配置**:
    *   残った商品を、左詰めルールに従って各段に再配置する（positionXの再計算）。
5.  **警告生成**: カットされた商品リスト、フェイス削減された商品リストを `warnings` 配列に格納。

### 1.3 ロジックB: 拡張 (Expand)
店舗棚幅 > 標準棚幅 の場合に適用される拡張ロジック。売れ筋商品を拡充する。

**処理フロー** (`automationService.ts` / `applyRuleB`):
1.  **判定**: `targetWidth > standardWidth`
2.  **1stパス - トップ商品拡充**:
    *   売上ランク上位10商品（`EXPAND_TOP_RANK_LIMIT = 10`、ランク数値が小さい順）を抽出。
    *   対象商品のフェイス数を2倍にする（`EXPAND_MULTIPLIER = 2.0`）。
    *   総幅を再計算し、`targetWidth` を超えた場合は追加を取り消して終了。
3.  **2ndパス - 準トップ商品拡充**:
    *   まだ余裕がある場合、残りの上位商品（11位以降）についてフェイス数を1.5倍にする（`EXPAND_SUB_MULTIPLIER = 1.5`）。
    *   同様に幅上限チェックを行う。
4.  **余白調整**:
    *   これ以上商品を増やせない場合、残りは「余白」として扱う（右端が空く形式）。

### 1.4 ロジックC: 同期 (Sync)
標準棚割の変更を、編集済みの個店棚割に反映させるマージロジック。

**処理フロー** (`automationService.ts` / `syncStorePlanogram`):
1.  **Inputs**: 最新の標準棚割（standardPlanogramId から取得）、現在の個店棚割
2.  **再生成**: 親の標準棚割を元にルールA/Bを再適用して個店棚割を再生成する。
3.  **ステータス更新**: `status = 'synced'`、`syncedAt` を現在日時に更新。

### 1.5 ロジックD: 商品配置の連結（Product Consolidation）
手動や自動処理において同じ商品が隣接した際に、物理的な1つの配置データ（Face数 > 1）へ統合するロジック。
*   **エディタでの適用**: ドラッグ操作等で同一段内で同じ商品を横にくっつけた場合、配置エンティティを一つにまとめ `faceCount` を加算する。
*   **デクリメント処理**: まとまったフェイスから手動で1つ削除した際には、`faceCount` を減算する。

### 1.6 フェイス数変更バリデーション
個店棚割エディタでフェイス数を増加する際の棚幅オーバーフローチェック:
1.  現在の棚段の全商品の合計幅を計算。
2.  増加分の幅（`product.width * (newFaceCount - currentFaceCount)`）を加算。
3.  合計が `planogram.width` を超える場合、アラートで残りスペースを表示して操作をブロック。

## 2. バッチ設計書 (Batch Design)

クライアントサイドでの「一括処理」として実装する。

### 2.1 個店棚割一括生成処理

*   **トリガー**: ユーザーがUI上の「自動棚割提案を作成」ボタンを押下。
*   **処理フロー** (`automationService.ts` / `batchGenerateStorePlanograms`):
    1.  選択された店舗リストを取得。
    2.  各店舗について：
        *   店舗の什器配置から棚幅を取得。
        *   ロジックAまたはBを適用して `StorePlanogram` データを生成。
        *   生成結果をSupabase（またはLocalStorage）に保存。
        *   進捗状況（`onProgress` コールバック）をUIに通知。
    3.  完了後、`GenerationResult[]` を返却（成功/警告/エラーの分類付き）。

## 3. 外部インタフェース設計書 (External Interface Design)

### 3.1 データ入出力 (Data I/O)

#### 3.1.1 商品マスタインポート
*   **形式**: Excel (.xlsx) / CSV
*   **文字コード**: UTF-8
*   **処理フロー**: ファイル読込 → プレビュー表示 → バリデーション → インポート実行
*   **項目**: JANコード, 商品名, カテゴリ, 幅(mm), 高さ(mm), 奥行(mm), 売上ランク, 組織階層各フィールド
*   **エラー処理**: JAN重複はスキップまたは上書き（選択可）。必須項目欠損はエラーログ出力。
*   **売上ランク自動計算**: `calculateSalesRank()` で売上金額から1-100のランクを自動付与。
*   **スキップ商品CSV出力**: インポートでスキップされた商品をCSVでエクスポート可能。

#### 3.1.2 商品階層インポート
*   **形式**: CSV
*   **項目**: 事業部CD/名, ディビジョンCD/名, ラインCD/名, 部門CD/名, カテゴリCD/名, サブカテゴリCD/名, セグメントCD/名, サブセグメントCD/名（16フィールド）

#### 3.1.3 商品CSVエクスポート
*   **形式**: CSV (Blob)
*   **関数**: `exportProductsToCSV(products)`
*   **項目**: 全商品フィールド

#### 3.1.4 Excelテンプレート生成
*   **形式**: Excel (.xlsx)
*   **関数**: `generateExcelTemplate()`
*   **用途**: インポート用の空テンプレートをダウンロード

### 3.2 アプリケーションインタフェース (Repository API)

## 4. 性能・運用設計書 (Performance & Operation Design)

### 4.1 性能要件
*   **レスポンス**:
    *   棚割エディタでのドラッグ＆ドロップ操作：**100ms以内** に追従すること。
    *   画面遷移：**1秒以内**。
*   **処理時間**:
    *   一括生成（100店舗想定）：**10秒以内**（クライアントサイド処理）。
*   **データ量**:
    *   商品マスタ：最大5,000件程度。
    *   1棚割あたりの商品配置数：最大300件程度。

### 4.2 運用・監視設計
*   **ログ**: アプリケーションエラーは `console.error` に出力。
*   **バックアップ**:
    *   Supabase: PostgreSQLの標準バックアップ機能を利用。
    *   LocalStorage: 「データエクスポート」機能によりJSONファイルとしてダウンロード可能。

## 5. 移行設計書 (Migration Design)

### 5.1 データ移行（初期ロード）
1.  **商品・店舗マスタ**: Excel/CSVフォーマットでデータを作成し、各マスタ画面のインポート機能から取り込む。
2.  **店舗マスタサンプル**: `seedData.ts` で18店舗のサンプルデータ（九州地方中心）を生成。
3.  **標準棚割**: マスタ移行後、本部担当者が画面上で手動作成する。

### 5.2 単位マイグレーション
*   システム初期は `cm` で設計されていたが、`20260227_cm_to_mm.sql` で `mm` に統一変換済み。
*   全テーブルの寸法カラムを×10倍に変換し、decimal精度を(5,2)→(8,2)に拡張。

## 6. テスト方針書 (Test Policy)

### 6.1 テストレベルと範囲

| テストレベル | 対象 | 実施方法 | 評価基準 |
| :--- | :--- | :--- | :--- |
| **単体テスト (Unit)** | 自動化ロジック (Cut/Expand/Sync) | Vitestによる自動テスト | 境界値で意図通り動作すること。ランク順序通りに処理されること。 |
| **結合テスト (Integration)** | 画面遷移、データ保存 | 手動操作 | 保存したデータが再読み込みで正しく復元されること。 |
| **システムテスト (E2E)** | 一連の業務フロー | ブラウザ手動操作 | マスタ登録～標準作成～個店展開～微調整～指示書出力の流れが滞りなく行えること。 |

### 6.2 重点テスト観点
1.  **ロジックの正確性**: カット時に高ランク商品が削除されていないか。拡張時に棚幅があふれていないか。
2.  **パフォーマンス**: 商品数が多い棚割（300品以上）での操作レスポンス。
3.  **フェイス数バリデーション**: 個店棚割でのフェイス数増加時に棚幅オーバーフローが正しく検出されるか。
4.  **例外系**: 極端に狭い店舗（商品が1つも入らない）での挙動。マスタ削除後の棚割データの整合性。
